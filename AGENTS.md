# Правила работы AI-агентов в Velvet

Эти правила обязательны для любой задачи, которая меняет код, миграции, конфигурацию, эксплуатацию или план проекта.

## Перед началом работы

1. Прочитать:
   - `docs/project_memory.md`;
   - `docs/development_status.md`;
   - `docs/stabilization_policy.md`;
   - последние относящиеся к задаче файлы из `docs/worklog/`;
   - предметную документацию затрагиваемого модуля.
2. Определить линию и фазу:
   - основное развитие Velvet Archive;
   - Velvet AI/Qwen;
   - исторический рефакторинг;
   - hotfix/эксплуатация вне фаз.
3. Создать новый файл `docs/worklog/YYYY-MM-DD-slug.md`.
4. До первого изменения production-кода заполнить раздел `Перед началом`:
   - цель;
   - исходный контекст;
   - планируемый объём;
   - критерии готовности;
   - риски и ограничения;
   - базовый commit и ветку.
5. Не начинать работу с неопределённым «продолжить фазу». Сначала указать точную линию, номер/название фазы и оставшийся срез.

## Предметная граница Velvet Archive

- этот репозиторий является архивным ботом для создателя и преимущественно единоличной эксплуатации;
- его штатные домены: персонажи, истории, медиа, референсы, публикации, аналитика, AI-проверки и эксплуатация;
- запрещено переносить сюда предметную логику другого аукционного бота: ставки, лоты, колоды аукциона, валюту, диапазоны ставок, победителей, типы и режимы аукционов;
- общие технические решения можно повторно использовать только как нейтральную infrastructure-идею, без импорта аукционных моделей, команд, таблиц и бизнес-правил;
- owner-oriented сценарий не следует искусственно усложнять многопользовательской ролевой моделью без реальной потребности продукта;
- перед использованием старого плана или контекста агент обязан сверять задачу с фактическим кодом текущего репозитория, а не с обсуждениями другого бота.

## Режим стабилизации

До закрытия стабилизационных ворот из `docs/stabilization_policy.md` приоритетом являются рефакторинг, производительность, надёжность, контроль, эксплуатация, тесты и документация существующих функций.

Новый код разрешён, если он:

- ускоряет или упрощает существующий сценарий;
- повышает надёжность, наблюдаемость или управляемость;
- уменьшает связанность и технический долг;
- добавляет внутренний repository/use case, кеш, очередь, batching, диагностику, метрики, staging, backup/restore automation или development tooling;
- улучшает UX уже существующей функции без расширения её предметного назначения.

Перед таким изменением worklog обязан ответить:

1. какую существующую функцию оно улучшает;
2. что конкретно станет быстрее, проще, надёжнее или понятнее;
3. почему это не новая предметная область;
4. как улучшение проверяется;
5. какие архитектурные границы сохраняются.

Нельзя добавлять новый пользовательский функционал только потому, что его технически возможно написать. Возможность компьютера терпеть ещё один handler не является продуктовой потребностью.

## Во время работы

- фактический код и слитые PR имеют приоритет над старым планом;
- старые применённые миграции не редактируются;
- handler не должен получать новый SQL;
- новая бизнес-операция создаётся через use case/domain service;
- новая команда получает кнопку или явный статус аварийной команды;
- опасные операции требуют подтверждения;
- временные patch/workflow-файлы удаляются из итогового diff;
- новые обнаруженные обязательства записываются в текущую сессию, а не оставляются только в чате;
- изменение фаз, приоритетов или архитектурного долга обновляет `docs/project_memory.md` и `docs/development_status.md`.

## После завершения работы

До слияния PR заполнить раздел `После завершения`:

- фактически сделано;
- изменённые модули и контракты;
- миграции и совместимость;
- точные проверки и их результат;
- PR и commit;
- незавершённое;
- следующий конкретный шаг.

Статус записи должен быть одним из:

- `завершено`;
- `частично`;
- `заблокировано`.

Статус `завершено` запрещён, если обязательная проверка не выполнена. Живая эксплуатационная проверка, которую CI не способен выполнить, записывается как отдельное незавершённое обязательство.

## Достоверность

- не писать «всё исправлено» без перечисления проверок;
- не писать «тесты зелёные» без команды или номера CI run;
- не скрывать rollback, ограничения среды и непроверенные Windows/Telegram сценарии;
- не записывать токены, пароли, cookies, private keys и строки подключения;
- не удалять историю worklog;
- исправление старой записи оформляется примечанием с датой;
- не смешивать разные линии фаз под одним номером.

## Pull request

Каждый содержательный PR должен:

1. изменять или добавлять хотя бы один файл `docs/worklog/*.md`, кроме `README.md`;
2. ссылаться на ID рабочей сессии;
3. содержать план, результат, проверки, остаток и следующий шаг;
4. иметь завершённую запись worklog до слияния;
5. проходить автоматический `project notes contract`.

## Исключения

Исключение допускается только для автоматического release/tag события, которое не создаёт PR и не изменяет исходный код. Срочный hotfix всё равно получает стартовую и итоговую запись, даже если обе добавлены в одном commit.
