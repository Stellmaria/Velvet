# Krita watermark bridge

## Назначение

Контур автоматизирует существующую подготовку изображений Velvet Archive к публикации. Владелец отправляет изображение боту, выбирает положение, цвет, прозрачность, размер и отступ, получает preview и подтверждает отдельный финальный файл. Исходник никогда не перезаписывается.

## Архитектура

```text
Telegram owner UI
    ↓
WatermarkService + PostgreSQL revisions
    ↓
локальный файловый bridge
    ↓
Krita Python plugin
    ↓
PNG output + JSON response
    ↓
Telegram preview / final document
```

Сетевой сервер не открывается. Бот и Krita работают с одним локальным каталогом:

```text
VelvetKritaBridge/
├── sources/
├── requests/
├── responses/
├── outputs/
└── previews/
```

Пути из request проверяются плагином и обязаны находиться внутри bridge-root.

## Установка плагина

1. Выполнить:

```powershell
python tools/krita/package_plugin.py
```

2. Импортировать созданный ZIP через управление ресурсами Krita либо скопировать `velvet_logo.desktop` и каталог `velvet_logo` в `pykrita`.
3. Включить `Velvet Anatomy Logo` в настройках Python-плагинов.
4. Перезапустить Krita.
5. В Krita выбрать `Инструменты → Сценарии → Velvet Anatomy: настроить bridge-каталог…`.
6. Указать тот же каталог, который задан в `.env` бота.

## Конфигурация бота

```env
KRITA_WATERMARK_ENABLED=true
KRITA_BRIDGE_DIR=C:\Users\<user>\VelvetKritaBridge
```

При выключенном флаге worker не регистрируется, а UI сообщает, что bridge отключён.

## Использование

1. Открыть `/menu`.
2. Нажать `💧 Водяной знак`.
3. Ответить изображением на форму.
4. Дождаться preview.
5. Кнопками изменить угол, цвет, прозрачность, размер или отступ.
6. `↩️ Предыдущая версия` создаёт новую revision с прошлыми настройками.
7. `🚫 Без знака` строит preview из исходника без watermark.
8. `✅ Сохранить` отправляет финальный PNG как документ.

Аварийный резерв: ответить командой `/watermark` на изображение.

## Состояния и восстановление

- `pending`: revision ждёт worker;
- `processing`: request передан Krita;
- `ready`: output получен;
- `error`: Krita или bridge вернули ошибку.

Каждая настройка создаёт новую revision. Если старая revision завершилась после новой, её результат сохраняется в истории, но не заменяет актуальный preview. После перезапуска worker продолжает проверять `processing` revisions и готовые response-файлы.

## Безопасность

- bridge не открывает TCP-порт;
- source, request, response и output обязаны находиться внутри одного разрешённого каталога;
- request создаётся атомарно через временный файл;
- Krita открывает копию исходника и экспортирует отдельный output;
- исходник не сохраняется поверх себя;
- пользовательский callback не содержит файловых путей или произвольных команд.

## Живая проверка Windows

CI может проверить Python-контракт, миграцию, callback и файловый protocol, но не может доказать реальный Krita API. Перед production-статусом обязательны:

1. Krita запущена с включённым плагином;
2. бот и Krita используют один bridge-каталог;
3. фото создаёт request и preview;
4. четыре угла, белый, чёрный, auto и HEX дают новые revisions;
5. откат и `Без знака` строятся из исходника;
6. финал отправляется документом;
7. перезапуск бота не теряет processing job.
