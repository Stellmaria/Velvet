# Память проекта Velvet

Дата актуализации: 17 июля 2026 года.

Этот документ нужен для непрерывности разработки. Он фиксирует фактическое состояние `main`, разделяет активные линии работы и не позволяет контексту другого проекта случайно стать «требованием» текущего репозитория. Люди почему-то особенно охотно смешивают системы, когда обе написаны на Python.

## Источники истины

- фактическое состояние продукта: `docs/development_status.md`;
- история заметных изменений: `CHANGELOG.md`;
- правила работы агента: `AGENTS.md`;
- записи каждой рабочей сессии: `docs/worklog/`;
- эксплуатация Supervisor: `docs/SUPERVISOR.md`;
- этот файл: карта фаз, решений, открытых обязательств и следующего направления.

При расхождении факты из кода, миграций, тестов и слитых PR имеют приоритет над старым планом.

## Предметная граница продукта

Velvet Archive является отдельным архивным ботом для создателя и преимущественно единоличного использования. Активные предметные области: персонажи, истории, медиа, референсы, публикации, аналитика, AI-проверки и эксплуатация владельцем.

Логика аукционного бота не относится к этому репозиторию. Ставки, лоты, аукционные колоды, валюты, диапазоны ставок, победители и режимы аукционов нельзя переносить в production-код, миграции или активный план Velvet Archive. Разрешено повторно использовать только нейтральные технические приёмы после адаптации к архивному домену.

# Линия A. Основное развитие текущего Velvet Archive

## Фазы 1–6. Функциональная основа

Статус: завершены.

1. **Аналитический центр**: канал, промты, персонажи, хэштеги, обсуждения, периоды и пагинация.
2. **Персонажи и классификация**: строгие алиасы, очередь нераспознанных тегов, ручная классификация и переиндексация.
3. **Публикации**: проверка постов, черновики, расписание, очередь, отправка и журнал событий.
4. **Качество данных**: perceptual hash, кандидаты дублей, обязательные поля, Telegram `file_id` и сиротские связи.
5. **Обсуждения и backup**: расширенная аналитика комментариев, рейтинги, активность, `pg_dump`, ротация и предмиграционная копия.
6. **Эксплуатационная устойчивость**: WorkerManager, системная диагностика, безопасное завершение ресурсов и repository foundation.

## Фаза 7. Модульная архитектура

Статус: основной перенос завершён.

- 7A: application layer и Telegram composition root;
- 7B: media quality repository/service;
- 7C: publication domain, validation, inbox и commands;
- 7D: characters, stories и references;
- 7E: archive, preview infrastructure и public archive;
- 7F: discussions, ingest, insights, rankings, activity и relink;
- 7G1: core config, access policy и Telegram middleware.

Исторические compatibility-фасады частично сохранены, но runtime monkeypatch больше не является штатной архитектурой.

## Фаза 18. Публичная граница PostgreSQL

- 18A: добавлен `Database.acquire()`;
- 18A: character и story repositories больше не используют приватный `_require_pool()`;
- 18B: archive и public archive repositories переведены на публичную границу;
- 18C: reference repository переведён с сохранением транзакций, пагинации и dedup по `file_unique_id`;
- 18D: media quality repository переведён с сохранением claim locks, fingerprint workflow, duplicate decisions и file checks;
- 18E: publication repository переведён с сохранением owner scope, пагинации, queue transitions, event logging и locked schedule query;
- 18F: discussion repository переведён с сохранением tracked-channel filters, parent lookup, reaction transaction, overview и participant stats;
- 18G: discussion ingest repository переведён с сохранением root resolution, alias lookup, транзакционного post/hashtag/link сохранения и thread matching;
- 18H: discussion insight repository переведён с сохранением CTE, period filters и derived summary metrics;
- 18I: discussion ranking repository переведён с сохранением общего `_rank_page`, нормализации пагинации и элементов `DashboardPage`;
- 18J: discussion activity repository переведён с сохранением silent-publication pagination, weekday/hour buckets и daily activity rows;
- SQL и транзакционные границы завершённых доменов сохранены;
- добавлен автоматический контракт, не допускающий аукционные доменные зависимости в production package;
- следующий изолированный срез: discussion post insight repository.

# Линия C. Исторический план раннего рефакторинга

Эта обязательная для старого CI секция больше не является активной линией Velvet Archive. Ранний план аукционного бота относится к другому продукту и исключён из выбора задач в этом репозитории. Его номера фаз, команды, таблицы и бизнес-правила не используются. При необходимости общая infrastructure-идея оценивается заново как отдельное техническое решение без аукционной предметной модели.

# Открытые обязательства

1. Один раз локально обновить старый Supervisor до commit `d49e69d` через `scripts/refresh_supervisor.ps1`, если это ещё не выполнено.
2. На целевой Windows проверить кнопку `Сам Supervisor → Перезапустить Supervisor`.
3. Проверить `Сам Supervisor → Обновить и перезапустить` на безопасном тестовом обновлении.
4. Подтвердить получение success/error отчёта bootstrap в Telegram.

# Правило выбора следующей задачи

Перед началом работы агент обязан:

1. определить линию и фазу;
2. проверить предметную границу Velvet Archive;
3. прочитать последний статус и незавершённые записи в `docs/worklog/`;
4. создать новую стартовую запись;
5. явно указать критерии готовности;
6. после реализации закрыть запись фактами, проверками, PR/commit и следующим шагом.

Новая работа не считается завершённой, пока её итог не записан в проект.
