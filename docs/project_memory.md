# Память проекта Velvet

Дата актуализации: 17 июля 2026 года.

Этот документ нужен для непрерывности разработки. Он отделяет параллельные линии фаз, фиксирует фактическое состояние `main` и не позволяет очередной «Фазе 2» затереть предыдущую «Фазу 2». Люди почему-то любят нумеровать всё с единицы заново.

## Источники истины

- фактическое состояние продукта: `docs/development_status.md`;
- история заметных изменений: `CHANGELOG.md`;
- правила работы агента: `AGENTS.md`;
- записи каждой рабочей сессии: `docs/worklog/`;
- эксплуатация Supervisor: `docs/SUPERVISOR.md`;
- этот файл: карта фаз, решений, открытых обязательств и следующего направления.

При расхождении факты из кода, миграций, тестов и слитых PR имеют приоритет над старым планом.

# Линия A. Основное развитие текущего Velvet Archive

## Фазы 1–6. Функциональная основа

Статус: завершены.

1. **Аналитический центр**: канал, промты, персонажи, хэштеги, обсуждения, периоды и пагинация.
2. **Персонажи и классификация**: строгие алиасы, очередь нераспознанных тегов, ручная классификация и переиндексация.
3. **Публикации**: проверка постов, черновики, расписание, очередь, отправка и журнал событий.
4. **Качество данных**: perceptual hash, кандидаты дублей, обязательные поля, Telegram `file_id` и сиротские связи.
5. **Обсуждения и backup**: расширенная аналитика комментариев, рейтинги, активность, `pg_dump`, ротация и предмиграционная копия.
6. **Эксплуатационная устойчивость**: WorkerManager, системная диагностика, безопасное завершение ресурсов и repository foundation.

## Фаза 7. Модульная архитектура

Статус: основной перенос завершён.

- 7A: application layer и Telegram composition root;
- 7B: media quality repository/service;
- 7C: publication domain, validation, inbox и commands;
- 7D: characters, stories и references;
- 7E: archive, preview infrastructure и public archive;
- 7F: discussions, ingest, insights, rankings, activity и relink;
- 7G1: core config, access policy и Telegram middleware.

Исторические compatibility-фасады частично сохранены, но runtime monkeypatch больше не является штатной архитектурой.

## Фаза 8. Supervisor и управление

Статус: завершена и затем расширена.

- внешний процесс Supervisor;
- restart, update, rollback и журналы;
- Codex worktree и подтверждённое применение;
- кнопочный центр владельца;
- подтверждение опасных операций.

## Фаза 9. Application use cases владельца

Статус: завершена.

Owner-операции вынесены из Telegram handlers в application use cases и boundary services: профили, истории, категории, вселенные, промты, алиасы, референсы, импорт и обсуждения.

## Фаза 10. Доступ и центр ошибок

Статус: завершена.

- точная матрица public/editor/owner;
- центр ошибок с группировкой, подтверждением и напоминаниями;
- разделение Supervisor controllers;
- защитные тесты команд и callback-префиксов.

## Фаза 11. Hardening и deployment

Статус: завершена.

- Python 3.13 и PostgreSQL 16;
- Dockerfile и Compose;
- container healthcheck;
- реальный backup restore drill;
- release workflow и стабильный релиз `1.3.0`;
- актуальная документация deployment и backup.

## Фазы 12–17. Архитектурная очистка P1

Статус: завершены.

- убран SQL и handler-to-handler imports из аналитики обсуждений;
- добавлен `PublicationActions`;
- разделены analytics management и owner reply-формы;
- multi-story перенесён в domain repositories;
- удалены runtime monkeypatch-мосты;
- compatibility installers превращены в безопасные фасады/no-op.

## Фаза 18. Публичная граница PostgreSQL

Статус: срезы 18A–18D завершены, P2-перенос продолжается.

- 18A: добавлен `Database.acquire()`;
- 18A: character и story repositories больше не используют приватный `_require_pool()`;
- 18B: archive и public archive repositories переведены на публичную границу;
- 18C: reference repository переведён с сохранением транзакций, пагинации и dedup по `file_unique_id`;
- 18D: media quality repository переведён с сохранением claim locks, fingerprint workflow, duplicate decisions и file checks;
- SQL и транзакционные границы завершённых доменов сохранены;
- следующий изолированный срез: publication repository.

## Фаза 19. Полный операционный контур Velvet AI

Статус: завершена.

- восстановлено полное меню качества;
- добавлен постоянный журнал AI-заданий;
- статусы `pending`, `processing`, `ready`, `error`, `interrupted`;
- шесть AI-потоков используют единый lifecycle;
- кнопочная форма сравнения с референсом;
- полное покрытие quality callback-тестами.

## Фаза 20. Удалённая эксплуатация Supervisor

Статус: код завершён, требуется первая живая проверка на целевой Windows.

- UTF-8-контракт дочернего процесса;
- фильтрация восстановимых Telegram polling disconnects;
- отпечаток запуска с путём и Git HEAD;
- безопасная удалённая консоль по allowlist;
- preview и одноразовое подтверждение команды;
- `shell=False`, таймауты, аудит и маскирование секретов;
- внешний Windows bootstrap для self-restart и self-update;
- fast-forward, тесты, rollback, lock и healthcheck;
- PR #95, commit `d49e69daa07e8654638015fde9944ad7d09003cb`.

# Линия C. Исторический план раннего рефакторинга

Это восстановленная линия из работ до текущей архитектуры репозитория. Она хранится для контекста и не является активной нумерацией `main`.

- Фаза 1: первоначальный аудит и разделение базовых обязанностей; точный объём частично остался только в старых рабочих архивах.
- Фаза 2: UID verification, HMAC/encrypted storage, expiry и аудит.
- Фаза 3: auction domain, ставки, правила и завершение аукционов.
- Фаза 4: standard/reverse/fast/free/black workflows, транзакции, locks и безопасная отмена.
- Фаза 5: transactional outbox, уведомления и перевод времени на UTC/`timestamptz`.
- Фаза 6: база данных, миграции и runtime/repository foundation.
- Фаза 7: обработка ошибок и дальнейшее модульное разнесение.
- Фаза 8: расширение тестов.
- Фаза 9: безопасность.
- Фаза 10: производительность.
- Фаза 11: CI и эксплуатация.

Часть целей этой линии позже была реализована уже под другой нумерацией в текущем Velvet Archive. Поэтому новые задачи нельзя называть просто «следующая фаза» без указания линии.

# Открытые обязательства

## Немедленная эксплуатационная проверка

1. Один раз локально обновить старый Supervisor до commit `d49e69d` через `scripts/refresh_supervisor.ps1`, если это ещё не выполнено.
2. На целевой Windows проверить кнопку `Сам Supervisor → Перезапустить Supervisor`.
3. Проверить `Сам Supervisor → Обновить и перезапустить` на безопасном тестовом обновлении.
4. Подтвердить получение success/error отчёта bootstrap в Telegram.

## P2

1. Продолжить Фазу 18 с `PublicationRepository`; characters, stories, archive, public archive, references и media quality уже используют `Database.acquire()`.
2. Сокращать широкие `except Exception` внутри бизнес-логики.
3. Добавить зашифрованную репликацию backup во внешнее хранилище.
4. Подготовить отдельную staging-конфигурацию и отдельного Telegram-бота.
5. Добавить метрики длительности AI-задач и стоимости внешнего provider.
6. Разделять крупные presentation-фасады только при реальном уменьшении связанности.

## P3

1. Добавить постепенный статический анализ типов.
2. Удалять неиспользуемые compatibility-фасады после проверки импортов.
3. Публиковать контейнерный образ в закрытый registry.
4. Расширить release notes ссылками на миграции и эксплуатационные изменения.

# Правило выбора следующей задачи

Перед началом работы агент обязан:

1. определить линию и фазу;
2. прочитать последний статус и незавершённые записи в `docs/worklog/`;
3. создать новую стартовую запись;
4. явно указать критерии готовности;
5. после реализации закрыть запись фактами, проверками, PR/commit и следующим шагом.

Новая работа не считается завершённой, пока её итог не записан в проект.
