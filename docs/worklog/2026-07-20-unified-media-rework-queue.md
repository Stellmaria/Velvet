- ID: `2026-07-20-unified-media-rework-queue`
- Линия/фаза: quality operations and rework workflow
- Статус: `завершено`

## Перед началом

### Цель

Создать единую очередь доработки для критичных и низкооценённых результатов Qwen и ручных решений администратора. Добавить отдельную Telegram-ветку, повторную проверку и финальное решение без дублирования записей.

### Исходный контекст

`media_ai_quality_checks` уже хранил вердикт Qwen и решения `accepted` / `fix_required`, но отдельного жизненного цикла доработки не существовало. Раздел `fix` был только фильтром по решению, не содержал статусов повторной проверки и истории действий.

### Планируемый объём

- отдельная таблица текущего состояния доработки;
- append-only история действий;
- автоматическое попадание критичных результатов Qwen и оценок ниже 70;
- синхронизация ручного `fix_required`;
- статусы `needs_fix`, `checking`, `ready_for_review`, `accepted`, `dismissed`;
- Telegram-команды `/rework`, `/reworks`, `/quality_rework`;
- карточки с превью, причиной, оценкой и действиями;
- повторная постановка Qwen-проверки;
- без изменений отложенного channel analytics.

### Критерии готовности

- один media_id имеет одну текущую запись доработки;
- Qwen и администратор объединяются в источник `mixed`, а не создают дубли;
- хороший повторный результат ждёт решения администратора;
- плохой повторный результат возвращается в `needs_fix`;
- принятие и снятие закрывают активную очередь;
- история решений сохраняется;
- CI и PostgreSQL integration tests проходят.

### Риски и ограничения

Миграция использует префикс `z001`, потому что ранее был занят терминальный числовой номер `999999`. Файл сортируется после всех существующих числовых миграций и не изменяет уже применённые SQL-файлы. Повторная проверка зависит от доступности Qwen worker.

## После завершения

### Фактически сделано

- добавлены `media_rework_items` и `media_rework_events`;
- добавлен PostgreSQL trigger `sync_media_rework_from_quality`;
- критичный вердикт или score ниже 70 автоматически создаёт/обновляет очередь;
- `fix_required` переводит запись в административную или смешанную доработку;
- повторная проверка сбрасывает quality check в `pending`;
- хороший повторный результат получает `ready_for_review`;
- плохой повторный результат снова получает `needs_fix`;
- Telegram-ветка показывает активную очередь и превью;
- добавлены действия «Принять», «Вернуть на проверку Qwen», «Снять»;
- новый контур регистрируется внутри существующего quality Router без дополнительного child Router;
- добавлены контрактные и PostgreSQL integration tests.

### Миграции и совместимость

Добавлена новая миграция `z001_unified_media_rework_queue.sql`. Она выполняет backfill уже существующих критичных результатов Qwen и ручных `fix_required`. Старые принятые решения не возвращаются в очередь.

### Проверки

Добавлены проверки SQL-контракта, Telegram registration boundary, объединения Qwen/admin в одну запись, повторной постановки анализа и ожидания ручного принятия после хорошего recheck. Полный GitHub CI запускается в PR.

### PR и commit

PR создаётся из ветки `agent/unified-media-rework-queue`; итоговый merge commit фиксируется после зелёного CI.

### Незавершённое

Отдельная кнопка очереди на главном экране quality dashboard не добавлялась: ветка доступна командами `/rework`, `/reworks`, `/quality_rework`. Массовые действия и назначение ответственного исполнителя остаются отдельным улучшением.

### Следующий шаг

После эксплуатационной проверки можно добавить счётчик очереди в dashboard и фильтры по источнику `Qwen / admin / mixed`, если объём активных записей станет большим.
