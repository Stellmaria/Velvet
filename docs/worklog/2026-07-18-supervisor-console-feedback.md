# Сессия: результат выполнения команд Supervisor в Telegram

- Дата: 2026-07-18
- ID: `2026-07-18-supervisor-console-feedback`
- Линия/фаза: стабилизация удалённого управления Supervisor
- Статус: завершено
- Ветка: `agent/supervisor-console-feedback`
- Базовый commit: актуальный `main` после слияния PR #121

## Перед началом

### Цель

Сделать выполнение безопасных консольных команд Supervisor понятным из Telegram: после подтверждения показывать запуск, автоматически дождаться завершения и вывести код возврата, длительность и очищенный вывод команды.

### Исходный контекст

Безопасная консоль уже использовала фиксированный allowlist, одноразовое подтверждение, запуск без shell и маскирование секретов. Supervisor сохранял результат выполнения в operation history, но Telegram после запуска показывал только сообщение «операция принята» и не отслеживал дальнейший статус.

### Планируемый объём

1. Добавить отдельное представление состояния консольной операции.
2. После запуска автоматически опрашивать operation history по ID.
3. Обновлять ту же Telegram-карточку при переходе в `running`, `success` или `error`.
4. Показывать код возврата, длительность, команду и безопасный хвост вывода.
5. Оставить ручное обновление и историю как резерв после перезапуска бота.
6. Не расширять список разрешённых команд и не добавлять shell-выполнение.

### Критерии готовности

- после подтверждения появляется явный статус запуска;
- итоговая карточка появляется автоматически без ручного перехода в историю;
- успешная команда показывает код `0`, длительность и вывод либо объяснение, что вывода нет;
- неуспешная команда показывает ненулевой код, вывод и доменную ошибку;
- длинный вывод ограничен лимитом Telegram и показывает хвост;
- существующий allowlist и запрет shell-конструкций не меняются;
- unit tests и проектный CI проходят.

### Риски и ограничения

- фоновое ожидание живёт внутри процесса бота и пропадает при его перезапуске;
- выполненная команда при этом не теряется, потому что Supervisor хранит operation history;
- для восстановления после перезапуска остаются ручная кнопка обновления и история;
- Telegram ограничивает длину сообщения, поэтому длинный stdout показывается с конца.

## После завершения

### Фактически сделано

- добавлен модуль `console_results.py` с карточками прогресса и результата;
- после `console.run` бот сразу показывает, что команда выполняется;
- создаётся контролируемая фоновая задача, которая раз в секунду запрашивает operation history и обновляет то же сообщение;
- итог показывает название, argv-команду, код возврата, длительность, stdout и ошибку;
- пустой stdout объясняется как нормальный результат для команд вроде `compileall` и `git fetch`;
- длинный stdout ограничивается безопасным хвостом;
- добавлена кнопка ручного обновления текущей операции;
- после завершения остаются переходы в историю, консоль, логи и главное меню Supervisor;
- список разрешённых команд, подтверждение и запуск без shell не изменялись.

### Миграции и совместимость

Миграции базы данных не требуются. Формат operation history не менялся: используются уже существующие поля `status`, `result.returncode`, `result.duration_seconds`, `result.command`, `result.output` и `error`.

### Проверки

- добавлены unit-тесты для состояний `running`, `success`, `error`, пустого и длинного вывода;
- проверяется наличие кода возврата и длительности в итоговой карточке;
- требуется GitHub CI после открытия PR.

Живая Telegram-проверка нужна после обновления рабочей Windows-установки: запустить `git status --short`, команду с пустым stdout и полный набор тестов.
