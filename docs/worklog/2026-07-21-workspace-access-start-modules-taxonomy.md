## Перед началом

### Цель

Продолжить реализацию «Моего пространства»: выдача Стэл права создать архив по Telegram ID, приватность новых архивов по умолчанию, выбор публичных архивов на `/start`, управляемые модули с отдельной справкой и собственные категории, вселенные и истории пользователя.

### Исходный контекст

PR #275 добавил workspace foundation, участников, роли, настройки и изоляцию персонажей. PR #276 перевёл приватный и публичный архив на workspace scope и добавил безопасный active workspace. Однако создание личного пространства ещё не было пользовательским продуктом: не существовало grant по Telegram ID, стартового выбора публичных архивов, отдельной политики модулей и собственного workspace-каталога категорий, вселенных и историй.

### Планируемый объём

- право создать личный архив выдаёт только global owner Стэл;
- кнопка создания показывается только пользователям с активным разрешением;
- все новые архивы приватны по умолчанию;
- системный `Velvet Anatomy` остаётся публичным;
- любой пользователь на `/start` видит переход к публичным архивам;
- владелец может включить публичный read-only просмотр своего архива;
- global owner задаёт разрешённые модули, владелец включает нужное подмножество;
- на экране каждого модуля есть кнопка справки;
- workspace может хранить собственные категории, вселенные и истории;
- в шаблонах доступно предложение импортировать вселенную КР из системного Velvet;
- существующий owner-контур Стэл не теряет Supervisor, System, Backup, Git/Codex и другие системные функции.

### Критерии готовности

- пользователь без grant не может создать архив даже прямым callback;
- пользователь с grant видит кнопку создания и может создать один личный workspace;
- созданный workspace имеет `public_archive_enabled = FALSE`;
- список публичных архивов показывает только включённые workspace;
- системный workspace ID `1` отображается публично;
- owner не может включить модуль, запрещённый global owner;
- категории, вселенные и истории уникальны только внутри workspace;
- импорт шаблона КР не изменяет системный каталог;
- tests, restore drill, type-check, Docker и notes зелёные.

### Риски и ограничения

Эта фаза создаёт рабочий product contract и Telegram entry/menu для workspace. Полное переключение старых character-management экранов с hardcoded constants на пользовательские taxonomy-записи остаётся следующим срезом. До него новые значения хранятся безопасно и управляются из «Моего пространства», но старые owner-экраны Стэл продолжают использовать системные каталоги.

## После завершения

### Фактически сделано

- добавлены grants создания архива, которые может выдать и отозвать только Telegram ID Стэл `7221553045`;
- реализованы команды `/workspace_grant`, `/workspace_revoke` и `/workspace_module`;
- `/start` для обычного пользователя показывает публичные архивы, а кнопку создания только при действующем grant;
- новый личный workspace создаётся приватным и сразу становится активным для владельца;
- `Velvet Anatomy` workspace ID `1` закреплён как публичный read-only архив;
- добавлен список публичных workspace и отдельный выбор просматриваемого архива;

- добавлены workspace-scoped категории, вселенные и истории;

- middleware допускает только подписанные workspace callbacks и активные workspace FSM-формы, а бизнес-операции повторно проверяют grant, membership, роль и module policy;

### Миграции и совместимость

Добавлена миграция `903_workspace_product_access.sql` с таблицами:

- `workspace_creation_grants`;
- `user_public_workspace_preferences`;
- `workspace_modules`;
- `workspace_categories`;
- `workspace_universes`;
- `workspace_stories`.

Миграция не меняет существующие character/media links. Существующий owner-контур Стэл продолжает работать без смены активных команд. Старые прямые вызовы публичного архива сохраняют workspace `1` по умолчанию.

### Незавершённое

- старые экраны создания и редактирования персонажей ещё не читают workspace taxonomy вместо системных hardcoded constants;
- custom `workspace_stories` ещё не используются существующим character story assignment flow;
- references, publications, Qwen и analytics требуют дальнейшего workspace wiring перед выдачей соответствующих модулей внешним владельцам;
- мастер подключения каналов пока хранит и защищает конфигурацию, но ещё не проверяет фактические Telegram admin rights пошаговым UI;
- интерфейс Стэл для выдачи grant и module policy пока представлен командами, а не отдельным многошаговым экраном.
