## Перед началом

### Цель

Убрать ложное предупреждение при невозможности удалить старое сообщение архивной ветки, изменить приоритет предложений Qwen-сетов с процента на количество материалов и сделать ручные решения по сетам взаимоисключающими.

### Исходный контекст

Telegram отвечает `message can't be deleted`, когда сообщение уже недоступно для удаления по возрасту или ограничениям чата. Архивный файл при этом корректно удаляется из базы, но Error Center создавал лишнее предупреждение. Предложения медиасетов сортировались только по `score DESC`, поэтому маленькая пара с высоким процентом показывалась выше крупной тематической группы. Один материал мог одновременно оставаться в нескольких pending-предложениях Qwen.

Кнопка `🎞 Один сет` в разделе дублей уже создавала pending-запись в общей таблице `media_set_candidates`, отмечала пару дублей как `ignored` и открывала предложение для ручного подтверждения. Отдельной отправки в Telegram-топик эта операция не выполняла.

### Планируемый объём

- считать `message can't be deleted` терминальным штатным результатом;
- ранжировать pending-сеты по числу доступных материалов, затем по score;
- не показывать предложения, в которых осталось меньше двух доступных материалов;
- при ручном переносе пары из дублей удалять эти материалы из остальных pending-предложений;
- при создании подтверждённого сета удалять его материалы из других предложений и закрывать опустевшие группы;
- добавить unit- и PostgreSQL-регрессии.

### Критерии готовности

- старое неудаляемое сообщение не создаёт WARNING;
- группа из большего числа изображений показывается выше маленькой группы с более высоким процентом;
- пара из дублей появляется в общем списке предложений сетов;
- подтверждённые или вручную объединённые материалы не остаются в конкурирующих Qwen-предложениях;
- предложения с нулём или одним доступным материалом скрыты;
- CI проходит.

### Риски и ограничения

Физически удалить старое сообщение Telegram всё равно невозможно, поэтому результат фиксируется как `not_deletable`, а удаление записи из базы продолжается. Галочки внутри открытого предложения остаются черновым выбором: исключение из других предложений применяется после создания сета либо после явного решения `🎞 Один сет` в дублях.

## После завершения

### Фактически сделано

- общий helper удаления сообщений распознаёт `message can't be deleted` и `message cannot be deleted` как терминальный результат без WARNING;
- добавлен repository списка кандидатов, который сортирует по `available_item_count DESC`, затем по `score DESC`;
- кандидаты с менее чем двумя доступными материалами не входят в выдачу;
- ручное решение `🎞 Один сет` сохраняет пару в общей очереди сетов, удаляет её материалы из других pending-предложений и закрывает опустевшие предложения;
- создание подтверждённого сета выполняется через repository в одной транзакции и удаляет выбранные материалы из остальных предложений;
- добавлены unit-тесты ранжирования и PostgreSQL-тесты взаимного исключения;
- architecture inventory обновлён до 117 корневых модулей.

### Миграции и совместимость

SQL-миграции не требуются. Используются существующие таблицы `media_set_candidates`, `media_set_candidate_items`, `media_sets`, `media_files` и `media_duplicate_candidates`. Callback-контракты и кнопки не меняются.

### Проверки

Добавлены `tests/test_media_set_candidate_policy.py`, `tests/test_media_set_exclusivity_postgres.py` и расширены тесты удаления сообщений и media-set actions repository. Полный GitHub CI запускается в PR.

### PR и commit

PR создаётся из ветки `agent/media-set-ranking-and-cleanup`; итоговый merge commit фиксируется после зелёного CI.

### Незавершённое

Реального отдельного Telegram-топика для предложений Qwen-сетов в текущем коде нет. Все предложения находятся в единой DB-очереди и открываются из раздела `🎞 Предложения медиасетов`.
