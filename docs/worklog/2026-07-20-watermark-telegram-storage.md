- Дата: 2026-07-20
- ID: `2026-07-20-watermark-telegram-storage`
- Линия/фаза: public archive storage and local cleanup
- Статус: `завершено`

- Базовый commit: `0ee46be3eb84756d887cb42ec730557c6afd64f1`

## Перед началом

### Цель

Перенести постоянное хранение всех одобренных watermark-файлов из локального каталога Krita в одну закрытую Telegram-ветку `-1004459280894 / 3`, сохранить быстрый индекс в PostgreSQL и удалять локальные рабочие файлы только после подтверждённой загрузки и записи метаданных.

### Исходный контекст

Fast-watermark сохранял полноразмерный PNG в `VelvetKritaBridge/outputs`, отправлял копию в административный чат и заменял `media_files.telegram_file_id`. Локальные source, output, preview, request и response не удалялись, поэтому каждый результат и каждая revision постепенно занимали место на компьютере.

### Планируемый объём

- единая Telegram-ветка без разделения по персонажам;
- отправка PNG документом без сжатия;
- имя файла с media_id, job_id, revision и SHA256;
- поисковая подпись с тегами, персонажами, исходным именем и шаблоном;
- сохранение chat_id, thread_id, message_id, file_id, file_unique_id, размера и SHA256;
- публичный архив использует file_id сообщения-хранилища;
- удаление локальных файлов только после успешной записи Telegram-ссылок в БД;
- удаление orphan-сообщения из Telegram, если архивная запись исчезла;
- без изменений отложенного channel analytics.

### Критерии готовности

- все новые одобренные архивные watermark-файлы идут в `-1004459280894`, thread `3`;
- файл можно найти по media_id, job_id, revision, SHA256, имени исходника или персонажу;
- база хранит прямую пару chat_id/message_id и Telegram file_id;
- в административный чат не отправляется дополнительная полноразмерная копия;
- локальные source/output/preview/request/response удаляются после успешного сохранения;
- при ошибке Telegram или БД локальный файл не теряется;
- CI и PostgreSQL migration contracts проходят.

### Риски и ограничения

Бот должен быть администратором или иметь право отправки документов в указанную форумную группу и ветку. Telegram является постоянным object storage, но PostgreSQL остаётся обязательным индексом: Bot API не предоставляет полноценный поиск сообщений по подписям. Удаление сообщения вручную из хранилища нарушит ссылку, поэтому такие удаления должны проходить через будущий audit/repair-контур.

## После завершения

### Фактически сделано

- добавлены defaults `WATERMARK_STORAGE_CHAT_ID=-1004459280894` и `WATERMARK_STORAGE_THREAD_ID=3`;
- создан сервис отправки, именования, SHA256, ссылок и локальной очистки;
- подпись содержит `#velvet_watermark`, `#media_<id>`, `#job_<id>`, `#rev_<n>`, персонажей, имя исходника, размер, SHA256 и шаблон;
- специализированный archive approval handler регистрируется перед общим watermark handler без нового child Router;
- archive approval загружает PNG только в Telegram-хранилище;
- `media_files.telegram_file_id` получает file_id из хранилища;
- после успешной записи удаляются source и все revisions текущего job из каталогов bridge;
- администратор получает короткое подтверждение и прямую ссылку на сообщение, но не вторую полноразмерную копию.

### Миграции и совместимость

Добавлена миграция `z002_watermark_telegram_storage.sql`. Старые watermarked-файлы продолжают работать по существующему `telegram_file_id`; новые storage-поля для них остаются NULL до отдельного backfill. Новые подтверждения заполняют Telegram storage index полностью.

### Проверки

Добавлены contract-тесты defaults, имени файла, поисковой подписи, ссылки, изолированной очистки job, миграции, порядка регистрации обработчика и repository storage SQL. Обновлён существующий fast-watermark repository test. Полный GitHub CI запускается в PR.

### PR и commit

PR создаётся из ветки `agent/watermark-telegram-storage`; итоговый merge commit фиксируется после зелёного CI.

### Незавершённое

Старые уже одобренные watermark-файлы автоматически не копируются в новую ветку. Для них нужен отдельный безопасный backfill по сохранённым Telegram file_id. Проверка существования storage-сообщений и восстановление удалённых сообщений остаются следующим эксплуатационным улучшением.
