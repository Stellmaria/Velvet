## Перед началом

### Цель

Не отправлять владельцу автоматический диагностический ZIP из-за одного кратковременного сбоя Telegram API и перенести на актуальный `main` оставшуюся полезную миграцию из конфликтующей ветки PR #291.

### Исходный контекст

Диагностический monitor считал один неуспешный `bot.get_me()` достаточным основанием для автоматической тревоги, хотя aiogram polling уже самостоятельно восстанавливал соединение. Ветка PR #291 одновременно отстала от `main` и конфликтовала с более новым guided onboarding, но содержала полезную миграцию каскадного удаления workspace-данных.

### Планируемый объём

- добавить накопительный счётчик последовательных Telegram health failures;
- отправлять Telegram-тревогу только после трёх неуспешных проверок подряд;
- сбрасывать счётчик после успешной проверки;
- сохранить немедленные тревоги для PostgreSQL, диска, workers, backup и CRITICAL-инцидентов;
- перенести только актуальную миграцию из PR #291;
- добавить regression tests и рабочую запись.

### Критерии готовности

- одна или две последовательные ошибки Telegram не отправляют ZIP;
- третья последовательная ошибка отправляет один ZIP;
- успешный probe обнуляет серию ошибок;
- остальные причины автоматической диагностики не получают задержку;
- migration `911_workspace_self_delete.sql` проходит migrator и CI;
- полный tests, type check, Docker, backup drill и project notes contract проходят.

### Риски и ограничения

Тревога о длительной недоступности Telegram появляется с задержкой в несколько циклов health monitor. Это намеренно: кратковременный сетевой сбой не должен выглядеть как авария. Миграция меняет только внешние ключи на каскадное удаление и не удаляет существующие данные сама по себе.

## После завершения

### Фактически сделано

- одиночный неуспешный Telegram health probe больше не создаёт автоматическую тревогу;
- тревога Telegram формируется только после трёх последовательных неуспешных проверок;
- успешная проверка сбрасывает накопленный счётчик;
- критические инциденты, PostgreSQL, диск, workers и backup продолжают запускать диагностику без задержки;
- добавлены regression tests для порога и сброса счётчика;
- перенесена миграция `911_workspace_self_delete.sql` с `ON DELETE CASCADE` для персонажей и watermark jobs;
- старый quick setup из PR #291 не переносился, потому что его уже заменил более новый guided onboarding в `main`;
- старое отдельное удаление пространства не переносилось, потому что актуальный owner controls уже содержит подтверждение и удаление.

### Миграции и совместимость

Добавлена additive migration `911_workspace_self_delete.sql`. Формат diagnostic bundle, schema Error Center и настройки `.env` не меняются. Миграция применяется обычным migrator перед polling.

### Проверки

Добавлены unit tests в `tests/test_owner_diagnostic_bundles.py`. После удаления одноразовых workflow выполнен отдельный чистый push для полного GitHub Actions CI.
