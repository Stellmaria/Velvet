## Перед началом

### Цель

Добавить обязательный мастер первого запуска личного пространства: объяснить владельцу устройство продукта, дать выбрать разрешённые модули, привязать реальные Telegram-чаты и темы, проверить права бота и показать итоговую готовность архива.

### Исходный контекст

Личное пространство уже имело роли, модульные политики, приватность, категории, вселенные, истории и таблицу `workspace_channels`. Однако после создания пользователь получал только сообщение о приватности. Состояние первоначальной настройки не сохранялось, темы Telegram-форума не фиксировались, а владелец не видел единого контрольного списка.

### Планируемый объём

- автоматически запускать мастер после ввода имени нового пространства;
- сохранять прогресс onboarding в PostgreSQL;
- показать интерактивный гид по модулям;
- использовать существующие allowed/enabled module policies;
- сохранять назначения chats/topics с `message_thread_id`;
- проверять, что бот является администратором и может публиковать;
- требовать `can_manage_topics` для веток персонажей в форумных чатах;
- зеркалировать совместимые назначения в `workspace_channels`;
- добавить команды повторного открытия, статуса, bind и unbind;
- не включать публичный архив автоматически.

### Критерии готовности

- новый архив сразу открывает мастер вместо тупикового сообщения;
- гид, выбор модулей, назначения и summary доступны кнопками;
- мастер можно закрыть и возобновить без потери прогресса;
- чат и текущая тема сохраняются по реальному Telegram update;
- права бота проверяются до сохранения назначения;
- включённые модули определяют обязательные назначения;
- завершение невозможно при неполной конфигурации;
- существующий workspace isolation сохраняется;
- полный CI, type check, Docker и project notes contract проходят.

### Риски и ограничения

Telegram не предоставляет безопасный общий список всех чатов пользователя. Поэтому для групп и форумных тем команда привязки выполняется внутри целевого чата/темы. Каналы без возможности отправить пользовательскую команду требуют отдельного адресного способа привязки по username/chat id; он должен оставаться под проверкой membership и прав бота. Системное пространство Velvet Anatomy не проходит пользовательский onboarding.

## После завершения

### Фактически сделано

- добавлена миграция `910_workspace_first_run_wizard.sql`;
- добавлены `workspace_onboarding` и `workspace_destinations`;
- сохранены status/current step, просмотр гида, подтверждение модулей и завершение;
- добавлен callback-driven мастер с intro, guide, modules, destinations и summary;
- первый ответ на `WorkspaceForm.waiting_workspace_name` перехватывается новым router до legacy handler;
- добавлены `/workspace_setup`, `/workspace_guide`, `/workspace_setup_status`, `/workspace_bind`, `/workspace_unbind`;
- сохраняются chat id, thread id, chat/topic titles, URL и снимок прав бота;
- назначения characters/public/publications/discussion/analytics/logs зеркалируются в существующие workspace channel kinds;
- readiness зависит от реально включённых модулей;
- публичность по-прежнему включается отдельно.

### Миграции и совместимость

Добавлена новая additive migration 910. Существующие таблицы и команды не переименовываются. Существующий `workspace_channels` остаётся источником для уже подключённых analytics/publication/public flows; новый слой хранит более точное назначение и тему форума. Новая форма имени зарегистрирована раньше legacy router, поэтому дублирующий старый handler остаётся fallback и не получает update.

### Проверки

Добавлен `tests/test_workspace_onboarding.py` с проверками обязательных назначений, readiness, схемы миграции, порядка router и команд. Полный тестовый workflow, type check, Docker build и project notes contract запускаются после открытия PR.

### PR и commit

Изменения подготовлены в ветке `agent/workspace-first-run-wizard`. Draft PR открывается в `main`; финальный merge commit фиксируется после зелёного CI и исправления найденных контрактных расхождений.

### Незавершённое

После merge требуется живая проверка в Telegram: создать тестовое личное пространство, привязать форумную тему персонажей, обычную тему материалов и канал публикаций, затем завершить checklist. Старые личные пространства получат status `not_started` и смогут открыть мастер вручную.

### Следующий шаг

После эксплуатационной проверки подключить `workspace_destinations.media` и `workspace_destinations.references` непосредственно к автоматическим маршрутам сохранения, если текущим продуктовым потокам потребуется отдельная тема помимо существующей workspace isolation. Затем добавить удобную адресную привязку закрытого канала по username/chat id из ЛС владельца.
