## Перед началом

### Цель

Перенести уже накопленные и будущие служебные артефакты с целевого Windows-компьютера в закрытый Telegram-форум `-1004459280894`, распределить их по указанным владельцем веткам, создать удобный PostgreSQL-индекс и удалять локальные копии только после подтверждённой загрузки.

### Исходный контекст

PR #241 уже направляет новые одобренные watermark-файлы в ветку `3`, но старые watermark, backup, завершённые Codex worktrees, логи, отчёты и release-архивы продолжали занимать локальный диск. Telegram Bot API не даёт полноценного серверного поиска по сообщениям, поэтому постоянное хранение должно сочетаться с отдельным индексом в PostgreSQL.

Реальная структура форума:

| Назначение | message_thread_id |
|---|---:|
| Watermarks Final | 3 |
| DB Backups Encrypted | 4 |
| Diagnostics Incidents | 9 |
| Exports Reports | 11 |
| Codex Patches | 7 |
| Releases Emergency | 13 |
| Rework Review | 15 |

### Планируемый объём

- общий индекс storage objects и multipart parts;
- первичный принудительный проход после обновления;
- периодическое досканирование новых артефактов;
- перенос старых watermarks по существующим Telegram `file_id`;
- шифрование backup до отправки;
- упаковка завершённых Codex tasks и удаление worktrees;
- перенос неактивных diagnostics, exports, reports и release archives;
- JSON-снимки очереди rework без повторного копирования медиа;
- поиск и скачивание через команды владельца.

### Критерии готовности

- используются реальные thread ID `3/4/9/11/7/13/15`;
- каждый объект получает logical key, SHA256, размер и Telegram message/file IDs;
- файлы выше лимита делятся на части;
- локальная копия удаляется только после записи всех частей в Telegram и PostgreSQL;
- повторный запуск не создаёт дубль, а завершает незавершённую очистку;
- `.env`, ключи, Git, виртуальные окружения, активные логи и активные Codex tasks не затрагиваются;
- CI, миграции и backup restore drill проходят.

### Риски и ограничения

Бот должен иметь право отправлять документы и удалять собственные сообщения во всех перечисленных ветках. Сам код не имеет удалённого доступа к диску Windows до обновления и запуска Supervisor, поэтому фактическая выгрузка происходит только на целевом компьютере. Telegram остаётся object storage, а PostgreSQL является обязательным индексом. Изменение секрета шифрования лишит возможности расшифровать старые backup.

## После завершения

### Фактически сделано

- добавлены `telegram_storage_objects`, `telegram_storage_parts` и журнал migration runs;
- настроены реальные ветки форума;
- реализован возобновляемый uploader с удалением orphan-сообщений;
- реализовано разбиение файлов на части до 45 MiB;
- первый проход запускается автоматически после применения миграции;
- повторные проходы выполняются каждый час и вручную через `/storage_migrate force`;
- добавлены `/storage`, `/storage_find` и `/storage_download`;
- старые одобренные watermarks копируются серверным Telegram `file_id`;
- завершённые Krita jobs очищаются после подтверждённого хранения;
- backup упаковываются, шифруются AES-256-GCM с ключом через scrypt, контрольным образом расшифровываются и сверяются по SHA256;
- завершённые Codex tasks архивируются вместе с prompt, diff, test output и git snapshot, затем worktree удаляется;
- diagnostics, exports, reports и releases выгружаются только после active-file grace period;
- очередь rework сохраняется JSON-снимком без повторной отправки медиа;
- команда `/storage` добавлена в меню владельца.

### Миграции и совместимость

Добавлена миграция `migrations/z003_telegram_storage_center.sql`. Таблица `backup_runs` получила ссылку на Telegram storage object и время offload. Существующие storage-поля PR #241 остаются совместимыми. Старые watermark продолжают работать во время backfill. Для шифрования добавлена зависимость `cryptography`.
