## Перед началом

### Цель

Устранить блокирующий доступ к быстрым действиям и мастеру личного архива: все workspace-scoped callback должны проходить middleware только после проверки active personal workspace, а начатые guided FSM-сессии не должны отбрасывать обычные сообщения и оставлять пользователя в подвешенном состоянии.

### Исходный контекст

В меню владельца уже есть `wsp:quick`, а guided actions и onboarding используют отдельные callback prefixes (`wqa`, `wob`). Access middleware пропускает лишь неполный набор workspace callback prefixes и распознаёт только исторический `WorkspaceForm` state. Из-за этого владелец личного архива может получить сообщение «служебная кнопка недоступна», а затем его ввод блокируется глобальной проверкой доступа вместо доставки активной форме.

### Планируемый объём

- составить единый явный allowlist tenant-aware callback prefixes;
- распознавать все рабочие personal workspace FSM-состояния без превращения произвольных FSM в обход owner policy;
- проверить handlers, callback actions, router order и кнопки start/onboarding/guided menu на тупики;
- сделать сообщения мастера понятнее: что создаётся, что где хранится, какие чаты/темы нужны и какие необязательны;
- добавить regression tests, выполнить полный набор проверок и создать атомарный commit.

### Критерии готовности

- owner личного пространства открывает быстрые действия, настройку, подключения, save picker и форму персонажа без global access denial;
- callback и message всегда повторно проверяют workspace membership/роль в их handler;
- кнопки «Назад»/«Отменить» очищают или корректно продолжают состояние;
- onboarding объясняет обязательный форум, персональные темы персонажей, хранение медиа и необязательные подключения;
- тесты покрывают prefix/state policy, кнопочные переходы и не допускают регрессии;
- commit создан только после зелёных проверок.

### Риски и ограничения

Это hotfix существующего UX и access boundary, не новая предметная область. Пропуск middleware ограничивается только известными workspace-prefixes и FSM-классами; конечные handlers по-прежнему обязаны проверять роль, module policy и принадлежность workspace. Живой Telegram/production database smoke невозможен без целевого окружения. Пользовательские незакоммиченные изменения должны быть проверены до commit и не могут быть перезаписаны.

### Обоснование стабилизации

Меняется существующий путь владельца в уже реализованный архив, мастер и операции сохранения. Он становится доступным, предсказуемым и безопасно отменяемым. Новая бизнес-область не вводится; Telegram controller не получает SQL, business/data checks остаются в текущих service/repository boundaries. Улучшение проверяется unit/integration contracts, статическим inventory, full tests и последующей живой проверкой.

## После завершения

### Фактически сделано

- Устранён отказ middleware для `wqa:*` и остальных tenant-aware callback; известные workspace FSM больше не блокируют владельца, но продолжают проверять активное членство.
- `/start`, возврат домой, быстрые действия и выбор персонажа очищают зависшее состояние; добавлена `/cancel`, завершающая форму и upload-сессию.
- Добавлен выбор между несколькими собственными и командными пространствами. Меню участника учитывает роль и показывает только допустимые разделы.
- Быстрые действия, модули аналитики, публикаций и Qwen ведут в реальные сценарии. Недоступные по конфигурации операции объясняют, какой модуль включить.
- Мастер теперь прямо объясняет обязательный форум персонажей, автоматическое создание персональной темы, хранение оригиналов и назначение каждого необязательного подключения.
- Создание персонажа из guided flow автоматически подготавливает его архивную тему; ручной ввод topic id убран.
- Привязка и отвязка назначения выполняются одной транзакцией для wizard-метаданных и runtime-канала. Привязать Telegram-чат/канал может только его администратор или владелец.
- Добавлены regression/contract tests для callback policy, FSM recovery, `/cancel`, start/router order, role-aware UI, реальных module entry points, автоматической темы и атомарной привязки.

### Изменённые модули и контракты

- `core/access` и Telegram access middleware: единый список workspace callback prefixes и явные workspace FSM prefixes.
- `domains/workspaces`: member-aware start state, безопасное чтение модулей участником, транзакционный контракт destination/channel.
- Telegram workspace routers: восстановление состояния, выбор пространства, guided actions, onboarding, owner/member controls и рабочие входы модулей.
- `workspace_ui.py`: start/selector/home клавиатуры с учётом роли и включённых модулей.
- Навигационный и P2 stability inventories обновлены по фактическому коду.

Новых миграций нет, применённые миграции не изменялись. Используются существующие таблицы `workspace_destinations` и `workspace_channels`; транзакционная запись совместима с текущими данными и сохраняет защиту от привязки одного чата к разным пространствам.
