## Перед началом

### Цель

Создать воспроизводимую инвентаризацию оставшихся production-вызовов `_require_pool()` и сформировать следующую очередь Фазы 18 по фактическому коду Velvet Archive.

### Исходный контекст

Фазы 18A–18L перевели четырнадцать repositories, включая полный discussion-контур, на публичный `Database.acquire()`. Текущий план больше не должен полагаться на ручной поиск или старую нумерацию другого проекта.

`Database._require_pool()` остаётся допустимой внутренней деталью самого класса базы, но domain, application, presentation и infrastructure clients не должны обращаться к ней напрямую без явно зафиксированного исключения.

### Планируемый объём

- просканировать весь production package `velvet_bot/`;
- найти вызовы и ссылки `_require_pool` с точными файлами и строками;
- отделить внутреннюю реализацию `Database` от внешних клиентов;
- классифицировать остаток по слоям и доменам;
- добавить самостоятельный скрипт инвентаризации;
- добавить CI/regression-тест, запрещающий новые внешние обращения;
- записать текущий allowlist или фактическую очередь без ложного объявления долга закрытым;
- обновить project memory, development status и changelog;
- определить следующий изолированный repository-срез.

### Критерии готовности

- отчёт строится автоматически из исходников, а не из вручную поддерживаемого списка строк;
- внутреннее определение и вызовы внутри `Database` классифицируются отдельно;
- каждый внешний production-вызов имеет путь, строку, слой и следующий action;
- CI падает при появлении нового неучтённого внешнего обращения;
- аукционная предметная модель не попадает в список задач;
- полный tests workflow, Docker build и project notes contract проходят;
- дневник закрыт фактическим результатом, PR/commit и следующим шагом.

### Риски и ограничения

- текстовый grep может дать ложные совпадения в комментариях и строках, поэтому основной анализ должен использовать AST;
- нельзя автоматически заменять найденные вызовы в этом PR;
- нельзя запрещать внутреннюю реализацию `_require_pool()` внутри `Database` до отдельного решения;
- dynamic access через `getattr` должен быть обнаружен или явно документирован;
- tests, migrations и docs не считаются production clients;
- следующий repository выбирается только после полной инвентаризации.
