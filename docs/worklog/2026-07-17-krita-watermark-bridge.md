## Перед началом

### Цель

Подключить существующий плагин Velvet Anatomy Logo для Krita к Velvet Archive как управляемый owner-сценарий нанесения водяного знака на медиа: выбор положения, цвета, прозрачности, размера и отступа, повторное построение preview из неизменяемого исходника, откат параметров и подтверждение финального результата.

### Исходный контекст

Velvet Archive уже хранит оригиналы и preview изображений, формирует публикации и имеет штатный WorkerManager, PostgreSQL repositories и owner-only кнопочное меню. Отдельный Krita-плагин уже умел вручную добавлять логотип в девяти положениях, менять белый/чёрный цвет, размер, отступ и непрозрачность, но не имел безопасного автоматического транспорта из Telegram, revisions, очереди и stale-result protection. Основная ветка параллельно продолжала Фазу 18, поэтому интеграция выделена в независимую ветку и не изменяет private-pool baseline.

### Какую существующую функцию улучшает изменение

Изменение улучшает существующие домены медиа и публикаций. Velvet уже хранит изображения, preview, оригиналы и готовит публикации; bridge автоматизирует промежуточное нанесение фирменного водяного знака вместо ручного открытия и настройки Krita для каждого изображения.

### Что станет проще и надёжнее

- параметры водяного знака задаются кнопками владельца;
- исходное изображение не перезаписывается;
- каждая новая настройка создаёт отдельную revision и preview;
- старые асинхронные результаты не могут заменить более новую revision;
- недоступная Krita сохраняет задание в контролируемом состоянии вместо потери операции;
- финальный экспорт остаётся отдельным артефактом.

### Почему это не новая предметная область

Watermark является частью подготовки уже существующих медиа к публикации. Новые персонажи, роли, аукционы или независимая пользовательская механика не добавляются.

### Планируемый объём

1. Добавить исходники адаптированного Krita-плагина в repository tooling.
2. Зафиксировать JSON protocol request/response и настройки watermark.
3. Добавить infrastructure bridge на локальной файловой очереди.
4. Добавить application service для revisions, preview и stale-result protection.
5. Добавить owner-only Telegram-интерфейс с кнопками положения, цвета, прозрачности, размера, отступа, отката, удаления и подтверждения.
6. Добавить persistence через новую миграцию и repository без SQL в handlers.
7. Подключить worker к штатному WorkerManager.
8. Добавить unit/regression-тесты, документацию и конфигурацию.

### Критерии готовности

- bridge protocol сериализуется и валидируется тестами;
- исходник и финальный/preview-файлы разделены;
- revision гарантирует игнорирование устаревшего ответа;
- handler не содержит SQL;
- owner-only callbacks имеют типизированный префикс и ранний acknowledgment;
- при недоступной Krita задание не теряется;
- плагин умеет удалить прежний watermark, применить настройки без диалога и экспортировать результат;
- существующие тесты и project notes contract проходят;
- живая проверка Krita на Windows явно отмечена как отдельное эксплуатационное обязательство, если недоступна в CI.

### Риски и ограничения

- Krita Python API и реальный запуск плагина нельзя полностью проверить в Linux CI;
- локальный bridge не должен открывать публичный сетевой порт;
- нельзя передавать произвольные команды или пути вне выделенного рабочего каталога;
- нельзя смешивать этот срез с продолжением Фазы 18;
- старые применённые миграции не изменяются;
- оригинал медиа никогда не перезаписывается.

## После завершения

### Фактически сделано

- добавлен owner-only раздел `💧 Водяной знак` в центре управления и аварийная команда `/watermark`;
- изображение загружается в выделенный bridge-каталог как неизменяемый исходник;
- реализованы четыре быстрых угла, белый, чёрный, auto и произвольный HEX, изменение прозрачности, размера и отступа;
- каждое изменение создаёт отдельную PostgreSQL revision;
- `↩️ Предыдущая версия` создаёт новую revision с предыдущими настройками;
- `🚫 Без знака` экспортирует отдельную версию из исходника без watermark;
- stale-result protection запрещает старому ответу Krita заменить более новый Telegram preview;
- финальный PNG отправляется владельцу отдельным Telegram-документом;
- локальный файловый bridge использует атомарные request/response JSON и не открывает сетевой порт;
- bridge-пути проверяются на принадлежность единому разрешённому каталогу;
- `krita-watermark` зарегистрирован в штатном WorkerManager только при `KRITA_WATERMARK_ENABLED=true`;
- исходники модифицированного Krita Python-плагина добавлены в `tools/krita/`;
- плагин автоматически опрашивает очередь, удаляет прежний vector layer, поддерживает HEX и анализ контраста, экспортирует отдельный output и закрывает открытый bridge-документ без сохранения исходника;
- добавлены упаковщик ZIP, эксплуатационная документация, `.env.example` и changelog;
- добавлены unit/regression-тесты protocol, settings bounds, path boundary и Telegram callback length;
- добавлен PostgreSQL integration test полного repository-сценария revisions → undo → ready → approve;
- установочный ZIP собран из финальных исходников плагина и проверен через `zipfile.testzip()`.

### Миграции и совместимость

Добавлена новая миграция `900_krita_watermark_bridge.sql` с таблицами `watermark_jobs` и `watermark_revisions`. Номера `099` и `100` оказались заняты существующими миграциями error incident center и AI quality checks; обе промежуточные версии были удалены до слияния. Применённые миграции не изменялись. Repository использует только публичный `Database.acquire()`. При выключенном `KRITA_WATERMARK_ENABLED` новый worker не регистрируется, существующие сценарии бота не меняются.
