## Перед началом

### Цель

Убрать падение Krita после установки plugin 2.1.0 и сохранить поддержку пользовательских SVG-логотипов без опасной растеризации через `PyQt5.QtSvg` внутри процесса Krita.

### Исходный контекст

После предыдущего исправления plugin импортировал `QSvgRenderer`, рендерил пользовательский SVG в PNG и затем вкладывал крупный base64 PNG в SVG, передаваемый `addShapesFromSvg`. Диагностические ZIP бота не содержат crash dump Krita, потому что падение происходит в отдельном desktop-процессе, но совпадает по времени с установкой plugin 2.1.0. Работники бота и очередь Krita в диагностике остаются healthy.

### Планируемый объём

- исключить `PyQt5.QtSvg`, `QSvgRenderer`, `QImage` и `QPainter` из plugin;
- преобразовывать пользовательский SVG в один плоский SVG-контур без вложенного `<svg>`;
- сохранять `defs`, стили, маски и ссылки на внутренние id;
- добавить защитный fallback при установке patch, чтобы ошибка Python-кода не блокировала запуск Krita;
- поднять версию plugin до 2.1.1;
- добавить regression contracts и инструкции переустановки.

### Критерии готовности

- plugin не импортирует `PyQt5.QtSvg`;
- пользовательский SVG не преобразуется в base64 PNG внутри Krita;
- итоговый документ не содержит вложенного пользовательского `<svg>`;
- Krita запускается даже при ошибке установки optional SVG patch;
- стандартный логотип и PNG-путь остаются совместимыми;
- CI проходит.

### Риски и ограничения

Диагностический bundle приложения не содержит Windows crash dump Krita, поэтому корневая причина desktop-crash определяется по последнему изменению plugin и устраняется консервативным исключением QtSvg/raster path. Очень сложные SVG с неподдерживаемыми Krita эффектами могут завершиться обычной ошибкой задания, но не должны обрушать приложение.
