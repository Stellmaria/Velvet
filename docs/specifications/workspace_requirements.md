# Моё пространство: требования и текущая реализация

> Статус: **официальное техническое задание проекта**.
>
> Этот документ является нормативным контрактом для разработки multi-workspace. При расхождении реализации, старых экранов или вспомогательной документации с данным ТЗ приоритет имеет этот документ. Изменения требований вносятся отдельным PR с явным описанием совместимости и миграций.

## 1. Общая модель

В боте поддерживаются отдельные пользовательские пространства с собственным архивом персонажей.

Каждое пространство изолировано от других по:

- персонажам;
- архивным материалам;
- категориям;
- вселенным;
- историям;
- настройкам публичности;
- доступным и включённым модулям;
- выбранному публичному архиву для просмотра.

Системное пространство `Velvet Anatomy` используется как основной архив Стэл и как источник готовых шаблонов.

---

## 2. Право на создание архива

Создать личный архив может только пользователь, которому Стэл выдала разрешение по Telegram ID.

Правила:

- право выдаёт только Стэл;
- разрешение хранится в базе;
- можно указать набор разрешённых модулей;
- можно задать максимальное число пространств;
- разрешение можно отозвать;
- прямой callback не позволяет обойти проверку;
- без разрешения кнопка создания архива на `/start` не отображается.

Команды Стэл:

```text
/workspace_grant TELEGRAM_ID
/workspace_grant TELEGRAM_ID module1,module2,module3
/workspace_revoke TELEGRAM_ID
```

Пример:

```text
/workspace_grant 123456789 characters,archive,taxonomy,references,public_archive
```

---

## 3. Экран `/start`

Обычный пользователь видит:

- `🌐 Посмотреть публичные архивы`.

Пользователь с активным разрешением дополнительно видит:

- `➕ Создать свой архив`.

Пользователь, у которого уже есть собственное пространство, дополнительно видит:

- `⚙️ Моё пространство`.

Кнопка создания появляется только после выдачи доступа Стэл.

---

## 4. Приватность и публичность

Все новые пользовательские архивы создаются приватными.

По умолчанию:

```text
public_archive_enabled = FALSE
```

Исключение:

- системный архив `Velvet Anatomy` сразу остаётся публичным.

Владелец личного архива может включить публичный режим.

Публичный режим является read-only:

- другие пользователи могут смотреть персонажей и материалы;
- они не могут редактировать архив;
- они не могут менять категории, вселенные, истории и настройки;
- владелец сохраняет полный контроль.

Если модуль публичного архива запрещён Стэл или выключен владельцем, пространство автоматически должно оставаться приватным.

---

## 5. Публичные архивы

На экране публичных архивов показываются только пространства, у которых включён публичный режим.

Пользователь может выбрать конкретный публичный архив.

Выбранное пространство применяется ко всему просмотру:

- категориям;
- вселенным;
- историям;
- персонажам;
- карточкам;
- материалам;
- лайкам;
- подпискам;
- просмотрам;
- скачиваниям.

Приватный архив нельзя выбрать через старую кнопку или прямой callback.

---

## 6. Модули пространства

Для каждого пространства существует отдельный набор модулей.

| Модуль | Назначение |
|---|---|
| `characters` | создание и управление персонажами |
| `archive` | хранение медиа персонажей |
| `taxonomy` | категории, вселенные и истории |
| `references` | хранение и сравнение референсов |
| `public_archive` | публичный read-only архив |
| `watermark` | версии материалов с водяным знаком |
| `qwen` | локальный AI-анализ |
| `publications` | подготовка публикаций |
| `analytics` | статистика и аналитика |
| `team` | роли и совместная работа |

Для каждого модуля хранятся два отдельных статуса:

```text
is_allowed
is_enabled
```

### `is_allowed`

Определяется Стэл.

Если модуль запрещён:

- владелец не может его включить;
- соответствующие кнопки скрываются;
- прямой callback блокируется.

### `is_enabled`

Определяется владельцем архива.

Владелец выбирает, какие из разрешённых модулей ему нужны.

Статусы в интерфейсе:

- `✅` модуль включён;
- `➖` модуль выключен владельцем;
- `⛔` модуль запрещён Стэл.

Команда Стэл для изменения разрешения уже созданного пространства:

```text
/workspace_module WORKSPACE_ID MODULE on
/workspace_module WORKSPACE_ID MODULE off
```

Пример:

```text
/workspace_module 7 qwen off
```

---

## 7. Справка по модулям

В меню каждого пространства есть кнопка справки.

У каждого модуля должна быть отдельная кнопка `ℹ️`.

Справка объясняет:

- назначение модуля;
- какие функции он добавляет;
- какие кнопки появляются;
- от чего зависит доступность;
- что может менять Стэл;
- что может менять владелец архива.

Кнопки разделов должны отображаться только тогда, когда соответствующий модуль:

```text
is_allowed = TRUE
is_enabled = TRUE
```

---

## 8. Собственные категории

Владелец архива может создавать собственные категории.

Категория содержит:

- внутренний ключ;
- название;
- emoji;
- порядок отображения;
- статус активности.

Пример формы:

```text
solo | Сольные персонажи | 🖤
```

Категории принадлежат конкретному workspace.

Одинаковый ключ может существовать в разных пространствах и не должен вызывать конфликт.

---

## 9. Собственные вселенные

Владелец архива может создавать собственные вселенные.

Вселенная содержит:

- внутренний ключ;
- название;
- emoji;
- обязательность выбора истории;
- порядок отображения;
- статус активности.

Пример формы:

```text
my_world | Моя вселенная | 🌒 | да
```

Последнее значение определяет, требуется ли для персонажей этой вселенной история.

---

## 10. Собственные истории

Владелец архива может создавать собственные истории внутри своей вселенной.

История содержит:

- ключ вселенной;
- ключ истории;
- короткое обозначение;
- полное название;
- порядок отображения;
- статус активности.

Пример формы:

```text
my_world | first_chapter | Г1 | Первая глава
```

Истории изолированы по workspace.

Одинаковые ключи историй в разных пользовательских пространствах не должны изменять друг друга.

---

## 11. Шаблон вселенной КР

В меню структуры пространства владельцу предлагается добавить готовую вселенную КР.

Кнопка:

```text
💎 Добавить вселенную КР
```

При импорте:

- создаётся вселенная `КР`;
- копируются истории из системного пространства `Velvet Anatomy`;
- копии принадлежат пользовательскому workspace;
- изменения пользователя не затрагивают системный каталог;
- повторный импорт не создаёт дубли.

Системный `Velvet Anatomy` используется только как источник шаблона.

---

## 12. Управление пространством

Раздел `Моё пространство` содержит:

- статус приватности или публичности;
- количество разрешённых модулей;
- количество включённых модулей;
- кнопку изменения публичности;
- список модулей;
- справку по каждому модулю;
- управление категориями;
- управление вселенными;
- управление историями;
- импорт шаблона КР.

---

## 13. Роли

Владелец личного архива получает роль:

```text
owner
```

Стэл остаётся global owner и может:

- выдавать право создания;
- отзывать право создания;
- ограничивать модули;
- управлять системным пространством;
- просматривать и администрировать пользовательские пространства;
- сохранять доступ к Supervisor, System, Backup, Git, Codex и другим системным функциям.

Будущие роли пространства:

- `owner`;
- `admin`;
- `editor`;
- `reviewer`;
- `viewer`.

---

## 14. Текущий технический срез

Реализовано:

- выдача разрешения по Telegram ID;
- отзыв разрешения;
- ограничение количества пространств;
- приватность новых архивов;
- публичный системный `Velvet Anatomy`;
- новый стартовый экран;
- список публичных пространств;
- выбор публичного workspace;
- workspace scope для каталога и медиа;
- отдельные разрешённые и включённые модули;
- справка по каждому модулю;
- собственные категории;
- собственные вселенные;
- собственные истории;
- импорт КР;
- PostgreSQL regression tests;
- dump/restore проверка новой схемы;
- type-check;
- Docker build;
- обновлённые архитектурные inventory-файлы.

---

## 15. Следующий этап

Следующим срезом необходимо перевести старые экраны управления персонажами с глобальных захардкоженных списков на workspace-каталог.

Нужно подключить workspace taxonomy к:

- созданию персонажа;
- редактированию категории;
- выбору вселенной;
- выбору истории;
- фильтрам каталога;
- сохранению медиа;
- референсам;
- публикациям;
- аналитике;
- ролям команды.

До завершения этого этапа новые категории, вселенные и истории уже безопасно хранятся и управляются в `Моём пространстве`, но часть старых административных экранов системного архива продолжает использовать прежние константы.
