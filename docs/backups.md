# Резервные копии Velvet

## Что резервируется

Команда `pg_dump` создаёт полную логическую копию рабочей PostgreSQL-базы. В архив попадают:

- персонажи, алиасы, вселенные и истории;
- связи персонажей с несколькими историями КР;
- медиа, Telegram `file_id`, preview и связи с архивом;
- промты, публикации, очередь и журнал отправки;
- решения по визуальным дублям, медиасеты и отчёты Velvet AI;
- аналитика канала и обсуждений;
- роли, настройки и служебные журналы.

Медиафайлы, существующие только на серверах Telegram и представленные одним `file_id`, физически не копируются. Локальные файлы вне PostgreSQL нужно резервировать отдельно.

## Требования

Нужны `pg_dump` и `pg_restore` той же или более новой основной версии PostgreSQL.

```env
BACKUP_DIR=backups
PG_DUMP_PATH=pg_dump
PG_RESTORE_PATH=pg_restore
```

На Windows допустимы полные пути:

```env
PG_DUMP_PATH=C:\Program Files\PostgreSQL\17\bin\pg_dump.exe
PG_RESTORE_PATH=C:\Program Files\PostgreSQL\17\bin\pg_restore.exe
```

## Кнопочный центр

Команда `/backup` открывает:

- **Создать копию сейчас**;
- **История копий**;
- **Проверить последнюю**;
- **Настройки**;
- **Очистить старые**.

По умолчанию ежедневная копия создаётся после 03:00, еженедельная после 04:00 в воскресенье, используется `Europe/Berlin`, хранится 14 последних файлов.

## Копия перед миграцией

Перед применением новых SQL-файлов бот:

1. определяет неприменённые миграции;
2. создаёт `pre_migration`-копию;
3. проверяет архив, список таблиц и SHA-256;
4. только после успешной проверки применяет миграции.

Если утилиты резервирования отсутствуют, обычный запуск без новых миграций продолжается с предупреждением. Если миграции действительно нужны, невалидная предмиграционная копия должна остановить изменение схемы.

## Быстрая проверка архива

Обычная проверка подтверждает:

- наличие и ненулевой размер dump;
- SHA-256;
- чтение через `pg_restore --list`;
- наличие ожидаемых таблиц;
- соответствие версии схемы.

Она обнаруживает повреждённый или явно неполный архив, но сама по себе не доказывает успешное восстановление данных.

## Restore drill

Для полной проверки добавлена команда:

```powershell
python scripts/backup_restore_drill.py --source-dsn $env:TEST_DATABASE_URL
```

Сценарий:

1. применяет текущие миграции к исходной тестовой БД;
2. добавляет уникальную контрольную запись;
3. создаёт custom-format dump;
4. создаёт новую временную PostgreSQL-базу;
5. полностью восстанавливает dump через `pg_restore --exit-on-error`;
6. повторно запускает мигратор на восстановленной базе;
7. проверяет контрольную запись;
8. сравнивает число миграций, персонажей и таблиц;
9. удаляет временную базу.

Опция `--keep-database` сохраняет временную базу для ручного расследования. Никогда не указывайте рабочую базу как временную цель.

GitHub Actions запускает restore drill при изменениях миграций и backup-кода, а также еженедельно по расписанию.

## Ручное восстановление

Автоматическое восстановление рабочей базы намеренно отсутствует. Для ручной проверки используйте отдельную пустую базу:

```powershell
createdb -U postgres velvet_restore_test
pg_restore `
  --dbname postgresql://postgres:PASSWORD@localhost:5432/velvet_restore_test `
  --clean --if-exists --no-owner --no-privileges `
  "backups\velvet_manual_YYYYMMDDTHHMMSSZ.dump"
```

После восстановления примените текущие миграции запуском бота или `Database.initialize()` в тестовом окружении, затем проверьте основные отчёты и связи.

## Внешнее хранение

Локальная ротация не защищает от поломки диска. Папку `BACKUP_DIR` следует копировать на отдельный носитель или в закрытое зашифрованное хранилище. Автоматическая удалённая репликация остаётся отдельной задачей развития.
